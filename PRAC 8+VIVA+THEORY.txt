BEFORE INSERT ON employees
FOR EACH ROW

    IF NEW.salary <= 0 THEN

VIVA

Viva Questions -

Experiment 8

Q: What is a trigger?

A stored SQL block that automatically fires when a specific DML event (INSERT, UPDATE, DELETE) occurs on a table. Cannot be called manually - invoked by the database engine.

Q: BEFORE vs AFTER trigger - when to use which?

→ BEFORE: for validation, preventing invalid data, modifying NEW values before saving. AFTER: for logging, audit trails, cascading changes to other tables after the operation completes.

Q: What are OLD and NEW in triggers?

NEW = new row data in INSERT/UPDATE. OLD = previous row data in UPDATE/DELETE. INSERT only has NEW. DELETE only has OLD. UPDATE has both OLD and NEW.

Q: How do you abort an operation inside a trigger?

Use SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'message'. This raises an error and cancels the DML operation. Only works in BEFORE triggers.

Q: Can you modify NEW values inside a trigger?

DBMS Practical - AISSMS College of Engineering, Pune

→ Yes - only in BEFORE triggers. SET NEW.column = value changes the value before it's saved to the table. In AFTER triggers, the row is already committed so NEW cannot be changed.

Q: Difference: trigger vs stored procedure?

Trigger: fires automatically on DML event, no parameters, cannot call manually. Procedure: called explicitly with CALL, accepts parameters, flexible execution.

Q: Can you use COMMIT or ROLLBACK inside a trigger?

→ No. Triggers run as part of the triggering transaction. Using COMMIT/ROLLBACK inside a trigger causes an error in MySQL.

Q: What is FOR EACH ROW?

→ MySQL only supports row-level triggers. FOR EACH ROW means the trigger fires once per affected row. E.g. if UPDATE affects 5 rows, the trigger fires 5 times.
